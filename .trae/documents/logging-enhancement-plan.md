# 增加日志记录详细过程，日志写入文件

## 概述
为 NyxAI 后端系统添加详细的日志记录功能，将日志写入文件，以便于调试、监控和问题追踪。

## 当前状态分析

### 现有日志系统
- 位置：`backend/app/utils/logger.py`
- 功能：基础日志配置，仅输出到控制台
- 使用位置：
  - `app/api/websocket.py` - WebSocket 连接管理
  - `app/services/llm.py` - LLM 服务调用
- 格式：`时间戳 | 日志级别 | 模块名 | 消息`

### 需要改进的方面
1. 日志仅输出到控制台，需要添加文件输出
2. 缺少日志轮转机制，可能导致日志文件过大
3. 部分关键流程缺少详细日志记录
4. 缺少结构化日志，难以进行日志分析

## 实施计划

### 阶段 1：增强日志配置系统

#### 1.1 更新配置文件
**文件：** `backend/app/config.py`

添加日志相关配置项：
- `LOG_DIR`: 日志文件存储目录（默认：`./logs`）
- `LOG_LEVEL`: 日志级别（默认：`INFO`）
- `LOG_FILE_MAX_BYTES`: 单个日志文件最大大小（默认：10MB）
- `LOG_FILE_BACKUP_COUNT`: 保留的日志文件备份数量（默认：5）
- `LOG_FORMAT`: 日志格式字符串
- `LOG_DATE_FORMAT`: 日志时间格式

#### 1.2 重构日志工具模块
**文件：** `backend/app/utils/logger.py`

改进内容：
1. 添加文件处理器（RotatingFileHandler）
2. 实现日志轮转机制
3. 支持多个日志文件（按模块分类）
4. 添加结构化日志支持（JSON 格式可选）
5. 改进日志格式，包含更多上下文信息：
   - 请求 ID（用于追踪请求链路）
   - 会话 ID（用于追踪会话）
   - 用户 ID（如适用）
   - 耗时（用于性能监控）

新功能：
- `setup_logging()`: 配置根日志处理器
- `get_logger()`: 获取指定模块的日志记录器
- `get_request_logger()`: 获取带请求上下文的日志记录器
- `add_request_context()`: 添加请求上下文信息

### 阶段 2：为关键流程添加详细日志

#### 2.1 Agent 执行流程
**文件：** `backend/app/agents/orchestrator.py`

添加日志记录点：
- Agent 启动：记录会话 ID、查询内容、启动时间
- Investigation 阶段：
  - 开始调查：记录调查参数
  - 调查完成：记录证据数量、耗时
  - 调查失败：记录错误详情
- Diagnosis 阶段：
  - 开始诊断：记录输入数据摘要
  - 诊断完成：记录诊断结果摘要、耗时
  - 诊断失败：记录错误详情
- Recovery 阶段：
  - 开始恢复：记录恢复计划摘要
  - 恢复完成：记录执行结果、耗时
  - 恢复失败：记录错误详情
- Agent 完成：记录总耗时、最终状态

**文件：** `backend/app/agents/investigation.py`, `diagnosis.py`, `recovery.py`

为每个 Agent 添加：
- 执行开始/结束日志
- 关键步骤日志
- 错误和异常日志（包含堆栈跟踪）
- 性能指标日志（耗时、资源使用）

#### 2.2 LLM 服务调用
**文件：** `backend/app/services/llm.py`

增强现有日志：
- 请求详情：
  - 完整的 system prompt（可配置是否记录）
  - 完整的 user prompt（可配置是否记录）
  - 模型参数（temperature, max_tokens 等）
  - 请求 ID
- 响应详情：
  - 完整响应内容（可配置是否记录）
  - Token 使用情况
  - 响应时间
  - 成本估算（如适用）
- 错误处理：
  - 重试次数
  - 错误详情
  - 超时信息

#### 2.3 WebSocket 连接管理
**文件：** `backend/app/api/websocket.py`

添加详细日志：
- 连接建立：
  - 会话 ID
  - 客户端信息（IP、User-Agent）
  - 连接时间
- 消息处理：
  - 消息类型
  - 消息内容摘要
  - 处理耗时
- 心跳机制：
  - 心跳发送/接收记录
  - 连接超时警告
- 连接断开：
  - 断开原因
  - 连接持续时间
  - 处理的消息总数
- 错误处理：
  - 异常详情
  - 重连尝试（如适用）

#### 2.4 API 端点
**文件：** `backend/app/api/*.py`

为所有 API 端点添加：
- 请求日志：
  - 请求方法、路径、参数
  - 请求头摘要
  - 请求体摘要
- 响应日志：
  - 响应状态码
  - 响应时间
  - 响应大小
- 错误日志：
  - 错误详情
  - 堆栈跟踪
  - 请求上下文

#### 2.5 数据库操作
**文件：** `backend/app/storage/repositories/*.py`

添加数据库操作日志：
- 查询操作：
  - 查询类型
  - 查询参数摘要
  - 返回记录数
  - 执行时间
- 写入操作：
  - 操作类型（INSERT/UPDATE/DELETE）
  - 影响的记录数
  - 执行时间
- 事务操作：
  - 事务开始/提交/回滚
  - 事务持续时间

#### 2.6 Skill 系统
**文件：** `backend/app/skills/*.py`

添加 Skill 相关日志：
- Skill 加载：
  - 加载的 Skill 列表
  - 加载耗时
  - 加载失败的 Skill 及原因
- Skill 执行：
  - Skill 名称、参数
  - 执行结果
  - 执行耗时

### 阶段 3：日志管理和维护

#### 3.1 创建日志目录结构
```
backend/logs/
├── app.log              # 应用主日志
├── agents.log           # Agent 执行日志
├── llm.log              # LLM 服务日志
├── websocket.log        # WebSocket 日志
├── api.log              # API 请求日志
├── database.log         # 数据库操作日志
└── errors.log           # 错误日志（ERROR 级别及以上）
```

#### 3.2 日志轮转配置
- 每个日志文件最大 10MB
- 保留最近 5 个备份文件
- 按日期归档旧日志

#### 3.3 日志清理策略
- 添加定期清理任务（可选）
- 清理 30 天前的日志文件
- 可配置保留天数

## 预期成果

1. **完整的日志系统**
   - 所有关键流程都有详细日志记录
   - 日志自动写入文件并轮转
   - 支持按模块分类的日志文件

2. **增强的可观测性**
   - 可以追踪完整的请求链路
   - 可以监控 Agent 执行过程
   - 可以分析 LLM 调用性能
   - 可以诊断 WebSocket 连接问题

3. **改进的调试能力**
   - 详细的错误日志和堆栈跟踪
   - 上下文信息丰富
   - 易于定位问题

4. **生产就绪**
   - 日志轮转防止磁盘空间耗尽
   - 性能影响可控
   - 符合安全和隐私要求

## 注意事项

1. **敏感信息保护**
   - 不记录 API 密钥、密码、token
   - 对用户数据进行脱敏处理
   - 可配置是否记录完整请求/响应内容

2. **性能考虑**
   - 异步日志写入（可选）
   - 控制日志级别，避免过多 DEBUG 日志
   - 使用高效的日志格式

3. **磁盘空间**
   - 合理配置日志文件大小和备份数量
   - 定期清理旧日志
   - 监控磁盘使用情况

4. **兼容性**
   - 保持现有 API 不变
   - 向后兼容现有日志调用
   - 渐进式迁移
